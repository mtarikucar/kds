## 3. [major] PATCH /tables/:id bypasses the active-order guard that PATCH /tables/:id/status enforces — operator can mark an occupied table AVAILABLE while an unpaid bill is still open

- **cluster:** tables-floorplan-qr
- **file:** `backend/src/modules/tables/tables.service.ts (304-342 (update); guard lives at 344-392 (updateStatus))`
- **symptom:** Reachable via the normal admin UI, not just a crafted request: TableManagementPage edit modal includes a status dropdown (TableManagementPage.tsx:92,99,109,132-135 statusOptions=[AVAILABLE,OCCUPIED,RESERVED] rendered with FormSelect) and submits via useUpdateTable -> PATCH /tables/:id (tablesApi.ts:49-71). An ADMIN/MANAGER opens the edit modal on an occupied table that has an unpaid order, picks 'Available', clicks save -> 'Table updated successfully' toast, table flips to AVAILABLE and can be re-seated while the prior bill is still open and unpaid. The one-tap card control on the SAME page correctly uses the guarded /status endpoint (TableManagementPage.tsx:372 updateTableStatus) and gets a 409 for the same action — so the protection visibly works in one place and is silently bypassed in another.
- **detail:** TablesService.updateStatus() (lines 344-392) is deliberately hardened: inside a transaction it counts active (non-PAID/non-CANCELLED) orders and throws ConflictException('Cannot mark table AVAILABLE while it has active orders') — the canonical gatekeeper for the documented invariant 'table status reflects real orders'. But TablesService.update() (lines 304-342) writes `data: updateTableDto` directly with NO active-order check. UpdateTableDto = PartialType(CreateTableDto), and CreateTableDto includes an `@IsEnum(TableStatus) @IsOptional() status` field (create-table.dto.ts:47-54), so `status` is a whitelisted, accepted property. The global ValidationPipe (main.ts:210-213) has whitelist:true but `status` is in the DTO so it is NOT stripped. Therefore `PATCH /tables/:id` with body `{ "status": "AVAILABLE" }` flips an OCCUPIED table to AVAILABLE while it still has open unpaid orders — exactly what updateStatus refuses.
- **suggested fix:** Stop update() from writing status, and route any status change through the guarded path. Two complementary fixes:

1) Strip status out of the generic update DTO so the only way to change status is the guarded endpoint. Cleanest: make UpdateTableDto omit status, e.g. in update-table.dto.ts:
   export class UpdateTableDto extends PartialType(OmitType(CreateTableDto, ['status'] as const)) {}
   (import OmitType from @nestjs/swagger). This keeps create defaulting AVAILABLE while removing status from the PATCH /tables/:id contract entirely, so ValidationPipe whitelist now strips an incoming status.

2) Defense-in-depth in TablesService.update(): if updateTableDto.status is present, delegate to/duplicate the updateStatus active-order guard (or simply forbid status here): e.g. at the top of update(), `if ('status' in updateTableDto) delete (updateTableDto as any).status;` OR run the same $transaction count check (status notIn [PAID, CANCELLED]) before allowing a status: AVAILABLE write.

Also update the frontend: remove the status FormSelect from the edit modal (TableManagementPage.tsx:455-459) and drop status from handleSubmit's submitData (line 109) so the modal only edits number/capacity/section; status changes already flow through the guarded one-tap control (line 372). Add a spec under the existing update() describe block asserting that update() with { status: AVAILABLE } either rejects or never writes status while active orders exist (mirroring the updateStatus test at spec lines 75-84).

## 4. [major] Self-pay (QR-menu diner) PayTR settlement has NO inquiry-recovery: a genuinely-paid intent that expires before its webhook lands is silently dropped — customer charged, order never settled, no refund

- **cluster:** paytr-settlement
- **file:** `backend/src/modules/customer-orders/services/self-pay-webhook.service.ts (71 (early-return on non-PENDING/non-PARTIALLY_SETTLED); self-pay-query.service.ts:231-240 (lazy expire); self-pay-sweeper.service.ts:32-41 (cron expire); subscription-scheduler.service.ts:562-566 (2nd cron expire) vs 604-663 (the recovery that exists ONLY for subscriptions))`

- **detail:** The customer self-pay (QR-menu) flow mints a PendingSelfPayment intent with a 15-minute TTL (INTENT_TTL_MINUTES, self-pay-pricing.util.ts:8) and a PayTR hosted-iframe charge. Settlement happens only via the SP-prefix branch of PaytrWebhookController -> SelfPayWebhookService.handleWebhookSuccess. That handler hard short-circuits when the intent is not PENDING/PARTIALLY_SETTLED (self-pay-webhook.service.ts:71). Meanwhile the intent is flipped to terminal EXPIRED purely on wall-clock time, with NO PayTR check, by three independent code paths: getPayStatus lazy-expire (self-pay-query.service.ts:231-240), the 5-min SelfPaySweeperService cron (self-pay-sweeper.service.ts:32-41), and the 30-min self-pay-orphan-cleanup cron (subscription-scheduler.service.ts:562-566). Consequence: a customer who completes payment near the TTL boundary, or whose success webhook is delayed/lost (endpoint downtime, IP-allowlist misconfig dropping callbacks per paytr-ip-allowlist.guard.ts, or PayTR retry latency) has their intent flipped to EXPIRED first; the genuine success webhook then arrives, hits the line-71 guard, and is silently dropped. The card was charged at PayTR (customer saw the success page), but payByItems never books a Payment row, the Order is never marked PAID, the table is never released, and NOTHING ever refunds or alerts — the SELF_PAY_AMOUNT_DRIFT / SELF_PAY_SETTLEMENT_FAILED reconciliation alerts only fire on the success/failure code paths that the EXPIRED short-circuit never enters. This is an ASYMMETRY: the subscription rail handles the identical race with handlePaytrPendingRecovery (subscription-scheduler.service.ts:604-663), an hourly cron that calls paytr.inquiryStatus(oid) for SubscriptionPayment rows PENDING >=2h and replays settlePayment, AND subscription PENDING payment rows are never TTL-swept to terminal before that window (paytr-orphan-cleanup only expires the Subscription/pendingPlanChange, not the SubscriptionPayment). The code comments prove the author missed this: subscription-scheduler.service.ts:601-602 claims SP rows are 'swept by a different cron' (implying equivalent handling), and self-pay-orphan-cleanup's own comment (lines 550-555) asserts 'EXPIRED is the truth (customer abandoned)' — but EXPIRED is set by wall-clock alone and is NOT verified against PayTR, so a real payment is indistinguishable from an abandonment. self-pay-reservation.service.ts:12 itself names the failure mode ('money taken on PayTR ... no auto-refund'). Operator/user-visible symptom: diner pays via QR self-pay, PayTR shows 'payment successful', but the bill still shows unpaid/remaining and the waiter is asked to collect again (or the table stays occupied); the diner is double-charged or charged-for-nothing with no automatic refund and no ops alert. Fix shape: mirror the subscription rail — before flipping a self-pay intent to EXPIRED, or on a hourly grace-window sweep, call paytr.inquiryStatus(merchantOid) for SP rows and replay handleWebhookSuccess when PayTR reports success; alternatively have handleWebhookSuccess also re-settle (not drop) an EXPIRED intent that PayTR confirms was paid.
- **suggested fix:** Mirror the subscription rail's recovery for self-pay. Two complementary changes:

(A) Make handleWebhookSuccess re-settle an EXPIRED intent that PayTR confirms paid, instead of silently dropping it. The webhook hash is already verified upstream (paytr-webhook.controller.ts:90-104), so a `status === "success"` callback is authoritative proof of charge. Change the line-71 guard so that an EXPIRED (or FAILED) intent on a SUCCESS webhook is NOT short-circuited: re-open it (status back to PENDING in a compound-WHERE updateMany, or branch into the settle loop directly) and run the existing pre-validate + payByItems path. payByItems idempotency (selfpay:<oid>:<orderId>) already prevents double-booking, so this is safe even against a concurrent retry. Keep terminal SUCCEEDED as the only true short-circuit.

(B) Add a PendingSelfPayment recovery cron paralleling handlePaytrPendingRecovery: hourly (advisory-locked), find PendingSelfPayment rows that are EXPIRED-with-failureReason in {expired, TTL expired (sweeper)} (or PENDING) within a grace window (e.g. created within last ~48h), call paytr.inquiryStatus(merchantOid) (adapter exists at paytr.adapter.ts:615), and on inquiry.status === "success" replay handleWebhookSuccess; cap rows/run like the subscription rail (take: 50). This catches both the TTL-race drop AND fully-lost callbacks. Emit a Sentry alert (e.g. SELF_PAY_RECOVERED / SELF_PAY_INQUIRY_PAID_BUT_EXPIRED) when a recovery books money on a previously-terminal row so ops sees the near-miss.

Optionally also: before the 5-min/30-min sweepers flip PENDING→EXPIRED, only do so for rows past TTL by a margin (e.g. expiresAt < now - 5min) to shrink the boundary race, since the authoritative recovery in (A)/(B) is the real fix.
